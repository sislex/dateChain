# dateChain — Пошаговый план реализации

Формат: **фазы → шаги**. У каждого шага — цель, задачи и **критерии приёмки (КП)**.
Прогресс каждого шага фиксируется в `PROGRESS.md` (статус, дата, коммит, заметки).
Правило: шаг не считается завершённым, пока не выполнены ВСЕ его КП (включая тесты и линт).

Легенда: 🟦 backend · 🟩 frontend · 🟨 UI-kit · ⬜ infra/общее · 🔒 core-flag монетизации отключён.

---

## ФАЗА 0 — Фундамент монорепо и CI ⬜

### Шаг 0.1 — Инициализация монорепо

- pnpm workspaces + Turborepo, структура `apps/*`, `packages/*`.
- Общие пресеты в `packages/config`: TS (strict), ESLint, Prettier, Jest.
- **КП:**
  - `pnpm install` проходит без ошибок.
  - `pnpm -w lint`, `pnpm -w typecheck` запускаются на пустых пакетах и проходят.
  - Turbo pipeline (`build/lint/test`) настроен, кэш работает.

### Шаг 0.2 — Docker-окружение

- `docker-compose.yml`: Postgres 16 (+PostGIS), Redis.
- `.env.example`, конфиг-загрузка.
- **КП:**
  - `docker compose up` поднимает Postgres+Redis, healthcheck зелёный.
  - Расширение PostGIS доступно в БД (`SELECT PostGIS_version()`).

### Шаг 0.3 — CI-пайплайн

- GitHub Actions (или аналог): install → lint → typecheck → unit → build.
- **КП:**
  - CI зелёный на пустом скелете.
  - Кэш зависимостей/turbo работает, прогон < разумного времени.

## ФАЗА 1 — UI-kit и дизайн-система 🟨

### Шаг 1.1 — Каркас `packages/ui` + Storybook

- Storybook 8, тема light/dark, дизайн-токены (цвета/градиент/шрифты из `03-DESIGN.md`).
- Настройка `@storybook/test-runner` (Playwright) в CI.
- **КП:**
  - `pnpm --filter ui storybook` открывает Storybook с токенами.
  - Тулбар переключает light/dark.
  - `pnpm --filter ui test-storybook` проходит на демо-истории в CI.

### Шаг 1.2 — Базовые компоненты

- Button (primary-градиент, варианты), IconButton (круглые action-кнопки), Input, TextArea,
  Select, Checkbox/Switch, Slider (радиус/возраст), Avatar, Badge, Tag/Chip, Spinner, Skeleton,
  Toast, Modal/BottomSheet, Tabs, BottomNav, SideNav.
- **КП:**
  - Каждый компонент имеет story со всеми состояниями и controls.
  - a11y: фокус-стили, ARIA, контраст (проверка `@storybook/addon-a11y`, без критичных ошибок).
  - Interaction-тесты (play) для интерактивных компонентов проходят.
  - Компоненты адаптивны (проверка в mobile/desktop viewport в Storybook).

### Шаг 1.3 — Доменные UI-компоненты Tinder

- Logo (пламя-градиент), SwipeCard, ActionBar, MatchScreen, PhotoPager, ProfileCard/ProfileDetails,
  ChatBubble, MatchListItem, TypingIndicator.
- Анимации на Framer Motion (см. `03-DESIGN.md`).
- **КП:**
  - SwipeCard в Storybook перетаскивается, показывает метки LIKE/NOPE/SUPER, «улетает» за порогом.
  - MatchScreen воспроизводит анимацию появления.
  - Interaction-тест свайпа (drag влево/вправо → корректный callback) проходит.
  - `prefers-reduced-motion` отключает анимации.

## ФАЗА 2 — Backend: фундамент 🟦

### Шаг 2.1 — Каркас NestJS + БД

- NestJS app, конфиг-модуль, TypeORM, подключение к Postgres, миграции, health-check.
- Testcontainers для интеграционных тестов БД.
- **КП:**
  - `GET /health` → 200 (проверяет БД и Redis).
  - Первая миграция применяется/откатывается.
  - Пустой e2e (Supertest) проходит в CI с поднятым Testcontainers Postgres.

### Шаг 2.2 — OpenAPI + генерация типов

- Swagger включён, схема экспортируется; генерация `packages/types` и `packages/api-client`.
- **КП:**
  - `/api/docs` открывается.
  - `pnpm gen:types` создаёт типы из OpenAPI; фронт импортирует их без ошибок TS.

### Шаг 2.3 — Auth-домен (OTP + JWT + RBAC) 🔒

- Сущности User/Device/Session, OTP (mock-провайдер SMS/email в dev), JWT access+refresh с ротацией,
  guard'ы, роли (user/admin/moderator/...), 2FA для админов (TOTP).
- **КП:**
  - Регистрация/логин по OTP выдаёт валидную пару токенов (unit + e2e).
  - Refresh ротирует токен; повторное использование старого refresh отклоняется.
  - Protected-эндпоинт без токена → 401, с ролью ниже требуемой → 403.
  - Покрытие auth-домена unit-тестами ≥ 85%.

## ФАЗА 3 — Backend: доменная логика 🟦

### Шаг 3.1 — Profile + Media

- CRUD профиля, атрибуты, интересы; загрузка фото на диск, `sharp` (ресайз/thumbnail/blur-hash),
  порядок фото, главное фото, защищённая отдача.
- **КП:**
  - Создание/обновление профиля валидируется (DTO), невалидные данные → 400.
  - Загрузка фото создаёт оригинал+thumbnail+blur-hash; чужое фото недоступно (403).
  - Переупорядочивание/удаление фото работает; unit + e2e зелёные.

### Шаг 3.2 — Discovery (гео + фильтры)

- Формирование колоды: PostGIS `ST_DWithin`, фильтры возраст/пол/радиус, исключение
  свайпнутых/заблокированных/себя, пагинация «колоды».
- **КП:**
  - Кандидаты в пределах радиуса и фильтров (unit на выборке фикстур).
  - Уже свайпнутые/заблокированные не возвращаются.
  - Производительность: запрос колоды использует гео-индекс (проверка `EXPLAIN`).

### Шаг 3.3 — Swipe + Match

- Запись свайпов (идемпотентно), детекция взаимного лайка → мэтч → доменное событие,
  дневные лимиты, rewind (флаг).
- **КП:**
  - Взаимный лайк создаёт ровно один мэтч (unit на гонке/дублях).
  - Превышение лимита лайков → 429/специальный ответ.
  - Событие «match» публикуется (проверка через мок шины/сокета).

### Шаг 3.4 — Chat + WebSocket

- Треды по мэтчам, сообщения (текст/изображение), статусы прочтения, пагинация истории.
- Socket.IO gateway: аутентификация сокета по JWT, rooms по мэтчу, typing, presence,
  Redis-adapter для масштабирования.
- **КП:**
  - Два клиента в одном мэтче: сообщение доставляется в реальном времени (e2e с socket.io-client).
  - Typing и presence эмитятся/сбрасываются корректно.
  - Неавторизованный сокет отклоняется; нельзя войти в чужую комнату (403).
  - История пагинируется; статусы «прочитано» обновляются.

### Шаг 3.5 — Moderation + Notifications + Admin API

- Жалобы, блокировки, очереди модерации, авто-флаги (эвристики), решения, аудит-лог.
- Внутренние уведомления + realtime-эмиты. Admin API: пользователи, метрики, feature-flags, конфиг.
- **КП:**
  - Жалоба создаёт запись в очереди; блокировка исключает пользователя из discovery/чата.
  - Действия админа пишутся в аудит-лог (кто/что/когда).
  - Метрики-эндпоинты возвращают агрегаты (unit на агрегаторах).
  - RBAC: модератор не может делать действия admin-уровня (403).

### Шаг 3.6 — Сиды и демо-данные

- Сид фейковых профилей (с фото-плейсхолдерами и гео) для дев/демо/e2e.
- **КП:**
  - `pnpm seed` наполняет БД N профилями; discovery возвращает непустую колоду.

## ФАЗА 4 — Frontend: пользовательское приложение 🟩

### Шаг 4.1 — Каркас `user-web`

- Vite + React + Redux Toolkit + RTK Query + Router; интеграция `@datechain/ui`;
  socket-middleware; layout (mobile bottom-nav / desktop 3-колонки).
- **КП:**
  - Приложение собирается и запускается; использует компоненты только из `@datechain/ui`.
  - Роутинг и защищённые роуты работают (редирект неавторизованного на онбординг).
  - Адаптив: mobile и desktop layout переключаются по breakpoint.

### Шаг 4.2 — Авторизация и онбординг

- Экраны приветствия, OTP-вход, многошаговый онбординг (A1), загрузка первых фото, гео-разрешение.
- **КП:**
  - Полный флоу регистрации до готового профиля проходит (Playwright e2e с mock/real API).
  - Валидация форм (zod) показывает ошибки; 18+ enforced.
  - Токены сохраняются, сессия восстанавливается после перезагрузки.

### Шаг 4.3 — Discovery / свайпы

- Колода на SwipeCard, ActionBar, жесты + кнопки, PhotoPager, детали профиля,
  MatchScreen, лимиты, пустое состояние.
- **КП:**
  - Свайп влево/вправо/вверх шлёт корректный запрос и убирает карту (оптимистично).
  - Взаимный лайк → показывается MatchScreen; данные согласуются с backend.
  - Пустая колода/лимит показывают корректные состояния.
  - Анимация свайпа соответствует `03-DESIGN.md` (визуально сверено).

### Шаг 4.4 — Фильтры подбора

- Экран Discovery Settings (радиус/возраст/пол/видимость), сохранение, влияние на колоду.
- **КП:**
  - Изменение фильтров перезапрашивает колоду с новыми параметрами.
  - «Скрыть меня» убирает из выдачи (проверка через backend).

### Шаг 4.5 — Мэтчи и чат (real-time)

- Список мэтчей + лента новых, чат (текст/изображение), статусы, typing, presence,
  unmatch, жалоба/блок, уведомления.
- **КП:**
  - Сообщения приходят в реальном времени между двумя сессиями (Playwright с двумя контекстами).
  - Typing/presence/статусы прочтения отображаются.
  - Unmatch удаляет диалог у обеих сторон; блок недоступен для чата.
  - Realtime-тост о новом мэтче/сообщении показывается.

### Шаг 4.6 — Профиль и настройки

- Редактирование профиля (фото drag&drop, био, интересы, атрибуты), предпросмотр,
  настройки (A6): уведомления, приватность, язык, блок-лист, пауза/удаление, выход.
- **КП:**
  - Изменения профиля сохраняются и отражаются в предпросмотре/колоде.
  - Пауза убирает из discovery; удаление аккаунта требует подтверждения и завершает сессию.
  - % заполнения профиля считается корректно.

## ФАЗА 5 — Frontend: админ-панель 🟩

### Шаг 5.1 — Каркас `admin-web` + вход

- Vite+React+RTK; вход email+пароль+2FA; RBAC-гейтинг маршрутов; layout (desktop-first + адаптив).
- **КП:**
  - Логин с 2FA работает; роли ограничивают доступ к разделам.
  - Использует только `@datechain/ui`.

### Шаг 5.2 — Дашборд + пользователи

- Дашборд-метрики/графики; список/поиск/карточка пользователя; действия (бан/заморозка/impersonate).
- **КП:**
  - Метрики отображаются из Admin API.
  - Действия над пользователем применяются и логируются в аудит (проверка через API).
  - Impersonate доступен только SUPER_ADMIN.

### Шаг 5.3 — Модерация + конфигурация

- Очередь жалоб, модерация фото/профилей, решения; управление feature-flags и лимитами; CMS-лайт.
- **КП:**
  - Обработка жалобы меняет её статус и применяет решение (бан/скрытие).
  - Переключение feature-flag влияет на поведение user-web (сквозная проверка).
  - Изменение лимита лайков отражается в backend-поведении.

## ФАЗА 6 — Сквозное качество, e2e и релиз ⬜

### Шаг 6.1 — E2E-сценарии (Playwright)

- Критичные пути: онбординг → свайп → мэтч → чат; жалоба → модерация → бан.
- **КП:**
  - Оба сквозных сценария зелёные в CI против поднятого стека (docker compose).

### Шаг 6.2 — Визуальная регрессия и a11y

- Storybook visual regression по всем компонентам; a11y-аудит ключевых экранов.
- **КП:**
  - Нет незапланированных визуальных диффов; нет критичных a11y-нарушений.

### Шаг 6.3 — Нагрузка, безопасность, наблюдаемость

- Rate-limit проверка, базовый security-review (OWASP), логи/метрики/health.
- **КП:**
  - Rate-limit срабатывает на превышении.
  - Security-review без критичных findings (auth, доступ к медиа, IDOR в чатах/профилях).

### Шаг 6.4 — Документация и релиз

- README, quickstart (`docker compose up` + `seed`), ADR, обновлён `PROGRESS.md`.
- **КП:**
  - Свежий клон разворачивается по инструкции и проходит demo-сценарий.
  - Все шаги в `PROGRESS.md` имеют финальный статус.
